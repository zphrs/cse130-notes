Security

Three Pillars

  • Confidentiality: only authorized users can access data
  • Integrity: data is not externally modified
  • Availability: data is available to authorized users when needed

Threat Models

Formalizes actions that an attacker can take. Assume that the attacker knows everything about the system.

Kerckhoffs’s Doctrine

The security of a system should depend on its key, not on its design remaining obscure.

Threat actor motivations

  • Cybercrime: profit
  • Nation-states: geopolitical gains/military
  • Hactivists: ideological principles
  • Thrill seekers: satisfaction/bragging rights
  • Insider threats: discontent / personal issues / negligence

Attacker Power

Assume an attacker can observe the timing/inputs/outputs and control the inputs for a single node system.

For a multi-node system, assume that the attacker can see/drop/change messages sent between nodes.

Heartbleed

The client could misreport the length of the message, causing the server to send back more data than it should. The solution was to ignore any requests that did not properly report the length of the message.

Side Channel Attacks

This includes timing attacks and cache attacks.

Password Checking

If you go one by one through the password and check if each character is correct, you can tell how many of the first characters are correct by the time it takes to return. This allows you to crack each character one by one by checking the time it takes to return.

Spectre and Meltdown

These are cache attacks that allow an attacker to read memory that they should not have access to by taking advantage of branch prediction and speculative execution when it comes to loading caches as the CPU only reverts pointers and not caching steps.

Dependency Attacks

SolarWinds

A vulnerability in the SolarWinds Orion platform allowed an attacker to insert a backdoor into the software that was then distributed to thousands of customers, making all of those customers vulnerable to the attacker, all because someone pushed their credentials to github.

Log4j

A vulnerability in the log4j library allowed an attacker to execute arbitrary code by sending a crafted message to a server that used the debug logging Java library.
