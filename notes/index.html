        A race condition is when the behavior of a system depends on the timing of events.

        Doesn’t necessarily mean a bug, might just mean a different result.

      • Mutual Exclusion (Mutex)
        Mutual Exclusion (Mutex)

        Ensures that only one thread is inside a critical section at a time (where the code reads and manipulates a shared variable).

        Shared variables can even include variables which are shared between parallel calls to the same function.
        • How to Use
          How to Use

          Call acquire before entering a critical section and call release after leaving the critical section. If there is a spinlock inside the critical section then make sure to regularly release and reacquire the lock during the spinlock.
      • Locks & Condition Variables
        Locks & Condition Variables

        Each condition variable has exactly one lock.

        Three functions:

        • wait() - release lock, wait for a signal/broadcast, reacquire lock
        • signal() wake any one of the threads that are waiting on the condition variable
        • broadcast() - wake all threads that are waiting on a condition variable
        • How to Use
          How to Use

          Use to avoid busy waiting. An example is to have a condition variable for when a buffer is full and another for when a buffer is empty.
      • Semaphores
        Semaphores

        Problem: previous solutions waste CPU times

        Uses two atomic operations:

        • up(S) <- S + 1
        • down(S): while (S <= 0) {}; S <- S - 1
          • Down can be modified to eliminate busy waiting for S to be greater than 0

        Note: up is sometimes V() and down is sometimes P().
        • Two Types
          Two Types

          Binary: 0 or 1 Counting: 0 to ∞
        • General Synchronization
          General Synchronization
          • Mutex
          • Rendezvous semaphore (one thread waits for another before proceeding)
          • Like rendezvous but n threads wait for the last one before proceeding
        • How to Use
          How to Use

          Use to avoid busy waiting. An example is to have a semaphore for when a buffer is full (initialized to N) and another for when a buffer is empty (initialized to 0). Lastly use a binary semaphore to protect the buffer (inc/dec).
        • Binary Semaphore vs Mutex
          Binary Semaphore vs Mutex

          A binary semaphore can be used to implement a mutex but a mutex cannot be used to implement a binary semaphore. A mutex must be released by the same thread that acquired it while a binary semaphore can be released by any thread.

      • Bounded Buffer Example
        Bounded Buffer Example 
        #define N ... // buffer size

Globals:
    Item buffer[N];
    int in = 0, out = 0, counter = 0;

void push(Item item) {
    while (counter == N) {} // busy waiting
    buffer[in] = item;
    in = (in + 1) % N;
    counter++;
}

Item pop() {
    while (counter == 0) {} // busy waiting
    Item item = buffer[out];
    out = (out + 1) % N;
    counter--;
    return item;
}

        The above code has a race condition.
        • Mutex
          Mutex

          One possible fix is with a mutex:

          Globals:
    Lock l;

void push(Item item) {
    l.acquire();
    while (counter == N) {
        l.release();
        l.acquire();
    } // busy waiting
    Buffer[in] = item;
    in = (in + 1) % N;

    counter++;
    l.release();
}

void pop() {
    l.acquire();
    while (counter == 0) {
        l.release();
        l.acquire();
    } // busy waiting
    Item item = buffer[out];
    out = (out + 1) % N;
    counter--;
    l.release();
    return item;
}
        • Condition Variable
          Condition Variable

          However, the mutex solution creates needless spin locks. A better solution is to use condition variables:

          Globals:
    Lock l;
    Condition full, empty;

void push(Item item) {
    l.acquire();
    while (counter == N) {
        full.wait(l);
    } // busy waiting
    Buffer[in] = item;
    in = (in + 1) % N;

    counter++;
    empty.signal(l);
    l.release();
}

void pop() {
    l.acquire();
    while (counter == 0) {
        empty.wait(l);
    } // busy waiting
    Item item = buffer[out];
    out = (out + 1) % N;
    counter--;
    full.signal();
    l.release();
    return item;
}

      • Strict Alternation
        Strict Alternation

        Taking turns by using a shared variable to keep track of whose turn it is

        Waiting thread spins on reading the shared variable, checking whether the shared variable is equal to its own ID
        • Downsides
          Downsides

          Only two threads allowed

          A thread busy doing noncritical work can prevent the other thread from entering the critical section

          Either thread can crash, causing the other to wait indefinitely
      • Dekker’s Algorithm
        Dekker’s Algorithm

        Track turn and have threads proclaim interest.

        Give mutex to a thread only if other thread is not interested and it’s the thread’s turn.

        Keep an array called wants_to_enter to track interest of the threads.

        Then have each thread wait until either it is their turn or the other thread is not interested. While a thread is waiting it sets wants_to_enter to false and then switches back to true once it’s their turn.
        • Code
          Code 
          bool wants_to_enter[2] = {false, false};
while wants_to_enter[other_id] {
    if turn != id {
        wants_to_enter[id] = false;
        while turn != id {
            // busy wait
        }
        wants_to_enter[id] = true;
    }
}

// critical section
turn = other_id;
wants_to_enter[id] = false;
// remainder section

          Supporting any number of threads is technically possible.

          No thread that is outside a critical reagion can block because a thread proclaims interest only when it is in the critical section.

          As long as the critical region eventually terminates, all threads get a turn.
      • Lamport’s Bakery Algorithm
        Lamport’s Bakery Algorithm

        Named after the bakery ticket system where you take a number and wait until your number is called.

        When a thread wants to lock, it looks at all other threads and takes a number that is one greater than the maximum number of all other threads. Then it waits until all other threads with a nonzero number less than its own have finished. To break ties, the thread with the lower ID goes first.

        There’s a problem though:

        m = max(thread_numbers);
my_number = m + 1;

        The above is not atomic. To solve this we keep an array of entering to track whether any thead is inside the above critical section. Then we wait to lock until both no one’s in the critical section and the above condition is true:

        lock(int id) {
    entering[id] = true;
    m = max(thread_numbers);
    my_number = m + 1;
    entering[id] = false;

    for (j in threads) {
        while (entering[j]) {
            // busy wait
        }
        while (thread_numbers[j] != 0
            && (
                thread_numbers[j] < my_number
                || (thread_numbers[j] == my_number && j < id)
            )
        ) {
            // busy wait
        }
    }
}

        Finally, this supports any number of threads, ensuring one thread at a time in the critical section and doesn’t require atomicity; but it does use busy waiting.

        Law of diminishing returns

        Q: How much will an optimization improves performance if only one stage of the system is affected?

        P = portion of time previously spent in the stage S = speedup of the stage

        This says that as the speedup of a stage increases, the overall speedup approaches:

        1 / (1 - P)

  • Day 23 - Caching
    Day 23 - Caching
    • Caching
      Caching

      Caching is used everywhere. CPU uses a cache to access memory faster, OS uses a file system cache, apps use cache to avoid refetching data, and modern systems almost always have a cache database (See Redis).
      • Memoization
        Memoization

        Cache the return value of each recursive call into a function.
        • Fibonacci Example
          Fibonacci Example 
          int fib(int n) {
  if (n < 2) return 1;
  if (cache[n] == 0) { // fill cache if empty
    cache[n] = fib(n-1) + fib(n-2);
  }
  return cache[n];
}

          Goes from O(2^n) to O(n).

          Side note: it’s possible to do fibonacci just using two variables and a loop. Look up “dynamic programming” to learn more.
      • Why does it work?
        Why does it work?

        Caching works on the assumption that a future request will be the same or at least similar to a past request and that the cost of preparing for future requests is less than the cost of recomputing the result.
      • Virtual Memory
        Virtual Memory
        • Caching Terms
          Caching Terms
          • Write-back (WB): writes occur only on cached data.
          • Write-through (WT): writes occur on both cache and memory.
          • Associativity
            • Fully associative: each item can be stored anywhere
            • Direct-mapped: each item can only be stored in one place
            • N-way: each item can be stored in N places
          • Mechanism: the way the cache is managed
          • Policy: the decision of what action to perform, especially when choosing which pages to evict.
        • CPU x OS
          CPU x OS

          The CPU and the OS collaborate to manage data access. The OS does the translation from virtual to physical memory for memory lookups. Then the CPU manages everything to do with the memory cache. For disk lookups the OS manages the disk cache.
      • Page Replacement
        Page Replacement

        Best possible algorithm is to always replace the page that will be used furthest in the future. This is impossible to know, so we use approximations based on past access patterns. Plenty of research has been done and is still being done on this topic.

        Note: Sometimes an OPTimal algorithm is used to compare other algorithms to understand how much optimization is possible and see if it’s worth it to further optimize.
        • FIFO
          FIFO
        • High Level
          High Level

          A page only needs to be written to disk if it was changed so keep track of whether a specific page was changed (i.e. dirtied). One bit per page is enough to track this.

          Four classes of pages:

          • 0: not referenced, not dirty
          • 1: not referenced, dirty
          • 2: referenced, not dirty
          • 3: referenced, dirty

          Then remove a page from the lowest non-empty class.
        • Implementation
          Implementation

          Maintain a queue of pages in the order which they entered memory. The page at the front of the list is replaced. This is good except for if the page in memory for the longest is used often. This is where the reference bit comes in.
          • Second Chance
            Second Chance

            Use the reference bit to give each page a second chance. If a referenced page is the oldest then it is moved to the back of the queue, set as unreferenced, and the next page is checked. If an unreferenced page is found then it is thrown out.

            The problem is that this algorithm might require a lot of time to find a page to evict.
        • Clock
          Clock

          Another way to model this is with a clock mentality. A clock hand points to the next page to replace and continues to move until it finds a page to replace, still using the reference bit to give pages a second chance. Same as second chance but with a circular list.
      • LRU
        LRU

        Keep counter in each page table entry. Each time a page is accessed, the counter is incremented. When a page needs to be replaced, the page with the lowest counter is replaced.

        The problem is that this is really expensive and is easily approximated by the clock algorithm.
        • NFU
          NFU

          Another good approximation of LRU is the Not Frequently Used algorithm.

          At each memory access, scan through page table. If R=1 for a page then add one to its counter value. Then on replacement pick the page with the lowest counter.

          The problem is that there’s no notion of age meaning that pages with high counter values will tend to keep them!
      • Testing
        Testing

        Start with a reference string which lists the pages that are accessed in order. Then run the algorithm on the reference string and measure the number of cache misses. You can use a table where the rows are the pages cached and the columns are the time steps.
        • Belady’s Anomaly
          Belady’s Anomaly

          The number of cache misses can increase when the cache size is increased. This motivated the study of paging algorithms.
