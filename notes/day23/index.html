Caching

Caching is used everywhere. CPU uses a cache to access memory faster, OS uses a file system cache, apps use cache to avoid refetching data, and modern systems almost always have a cache database (See Redis).

Memoization

Cache the return value of each recursive call into a function.

Fibonacci Example

int fib(int n) {
  if (n < 2) return 1;
  if (cache[n] == 0) { // fill cache if empty
    cache[n] = fib(n-1) + fib(n-2);
  }
  return cache[n];
}

Goes from O(2^n) to O(n).

Side note: it’s possible to do fibonacci just using two variables and a loop. Look up “dynamic programming” to learn more.

Why does it work?

Caching works on the assumption that a future request will be the same or at least similar to a past request and that the cost of preparing for future requests is less than the cost of recomputing the result.

Virtual Memory

Caching Terms

  • Write-back (WB): writes occur only on cached data.
  • Write-through (WT): writes occur on both cache and memory.
  • Associativity
    • Fully associative: each item can be stored anywhere
    • Direct-mapped: each item can only be stored in one place
    • N-way: each item can be stored in N places
  • Mechanism: the way the cache is managed
  • Policy: the decision of what action to perform, especially when choosing which pages to evict.

CPU x OS

The CPU and the OS collaborate to manage data access. The OS does the translation from virtual to physical memory for memory lookups. Then the CPU manages everything to do with the memory cache. For disk lookups the OS manages the disk cache.

Page Replacement

Best possible algorithm is to always replace the page that will be used furthest in the future. This is impossible to know, so we use approximations based on past access patterns. Plenty of research has been done and is still being done on this topic.

Note: Sometimes an OPTimal algorithm is used to compare other algorithms to understand how much optimization is possible and see if it’s worth it to further optimize.

FIFO

High Level

A page only needs to be written to disk if it was changed so keep track of whether a specific page was changed (i.e. dirtied). One bit per page is enough to track this.

Four classes of pages:

  • 0: not referenced, not dirty
  • 1: not referenced, dirty
  • 2: referenced, not dirty
  • 3: referenced, dirty

Then remove a page from the lowest non-empty class.

Implementation

Maintain a queue of pages in the order which they entered memory. The page at the front of the list is replaced. This is good except for if the page in memory for the longest is used often. This is where the reference bit comes in.

Second Chance

Use the reference bit to give each page a second chance. If a referenced page is the oldest then it is moved to the back of the queue, set as unreferenced, and the next page is checked. If an unreferenced page is found then it is thrown out.

The problem is that this algorithm might require a lot of time to find a page to evict.

Clock

Another way to model this is with a clock mentality. A clock hand points to the next page to replace and continues to move until it finds a page to replace, still using the reference bit to give pages a second chance. Same as second chance but with a circular list.

LRU

Keep counter in each page table entry. Each time a page is accessed, the counter is incremented. When a page needs to be replaced, the page with the lowest counter is replaced.

The problem is that this is really expensive and is easily approximated by the clock algorithm.

NFU

Another good approximation of LRU is the Not Frequently Used algorithm.

At each memory access, scan through page table. If R=1 for a page then add one to its counter value. Then on replacement pick the page with the lowest counter.

The problem is that there’s no notion of age meaning that pages with high counter values will tend to keep them!

Testing

Start with a reference string which lists the pages that are accessed in order. Then run the algorithm on the reference string and measure the number of cache misses. You can use a table where the rows are the pages cached and the columns are the time steps.

Belady’s Anomaly

The number of cache misses can increase when the cache size is increased. This motivated the study of paging algorithms.
